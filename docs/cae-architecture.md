# CAE-продукт «Диплом-Инж.CAE» — Архитектура и Техническое Задание

> **Версия:** 0.1 (черновик архитектуры, без кода)
> **Дата:** 2026-05-24
> **Стек целевой:** React + R3F + Vite на фронте, Python 3.11 Cloud Functions (NumPy/SciPy) на бэке, PostgreSQL для данных, S3 для проектов и отчётов
> **Базовая теория:** Timoshenko Beam FEM для 3D-рам (без STEP-импорта в MVP)

## 1. Бизнес-контекст и стратегия

### 1.1. Продуктовая линейка
Отдельный SaaS-продукт **«Диплом-Инж.CAE»** на поддомене `cae.диплом-инж.рф`. Не смешивается с основным маркетинговым сайтом, имеет собственный личный кабинет, биллинг и проекты пользователей. При этом обе платформы перекрёстно работают на лидогенерацию:

- В CAE есть кнопка «Нужна помощь наставника?» → форма заявки в основной сайт (лид в Битрикс24 + email).
- В основном сайте на странице тарифов и блога — баннер «Считай ВКР в нашем облачном CAE».

### 1.2. Аудитория и тарифы

| Тариф | Цена | Целевая аудитория | Лимиты |
|-------|------|-------------------|--------|
| **Демо (без регистрации)** | 0 ₽ | Холодный трафик: студенты, инженеры на пробе | 1 расчёт в день на IP+fingerprint, до 10 элементов, без сохранения, водяной знак в PDF |
| **Базовый** | 590 ₽/мес или 200 ₽ за разовый расчёт | Студенты ВКР | 5 проектов, до 50 элементов, эпюры, PDF без водяного знака |
| **Продвинутый** | 1490 ₽/мес | Аспиранты, частная практика | 50 проектов, до 500 элементов, экспорт в DXF/STEP-обзор, библиотека ГОСТ-профилей расширенная |
| **Профи** | 3490 ₽/мес | Действующие инженеры, малые КБ | Безлимит проектов, до 5000 элементов, командный доступ (до 3 чел.), приоритетный solver-pool, нелинейный расчёт (опц.) |

### 1.3. Защита бесплатного режима от обхода
- **Layered identity:** сочетание IP, FingerprintJS Pro, временного browser-cookie с server-signed JWT и hash локального storage.
- **Лимит**: один расчёт демо в сутки на сочетание `(ip_hash, fingerprint_hash)`. История попыток хранится в `demo_attempts` (PostgreSQL) c TTL 30 дней.
- **Защита от VPN:** проверка IP по базе известных VPN/proxy/Tor exit-нод (например, `ipapi.is`, `vpnapi.io` через бэкенд) — при детекте VPN расчёт идёт, но в шапке PDF пишется «без верификации» (мягкое давление).
- **Капча:** Yandex SmartCaptcha (российский провайдер, ровно ложится в стек) перед запуском solver-а в демо-режиме.
- **Rate-limit:** на уровне backend-функции — не более 3 попыток рассчитать за 5 минут с одного IP.

Эта связка не убьёт настоящих пользователей, но повысит цену обхода до уровня, когда проще купить подписку за 590 ₽.

### 2.2. Поток данных «от модели до отчёта»

1. **Пользователь** рисует раму на R3F-канве: точки → узлы (Nodes), линии → элементы (Elements). К узлам цепляются закрепления (BC), к элементам — материалы, сечения и нагрузки.
2. **Frontend** сериализует модель в JSON по схеме `SolverPayload` (см. §4.1) и валидирует локально (Zod-схема).
3. **POST `/cae-solver`** с JWT в заголовке. Backend:
   - Валидирует payload (Pydantic 2.x).
   - Проверяет квоту тарифа (число элементов, расчётов в день).
   - Запускает `TimoshenkoFrameSolver` (NumPy/SciPy).
   - Возвращает `SolverResponse` (см. §4.2).
4. **Frontend** строит эпюры в R3F (вспомогательные сцены для N, Q, M), деформированную форму (масштабируемая визуализация перемещений), карту σ_eq по Мизесу.
5. **Пользователь** отмечает чекбоксами разделы для отчёта → POST `/cae-report` → бэкенд рендерит HTML через Jinja2 и PDF через WeasyPrint, загружает в S3 → возвращает CDN-ссылку.

### 2.3. Личный кабинет и хранение проектов

- **Authentication:** email + пароль (Argon2id) + JWT (RS256, 1 час access + 30 дней refresh).
- **Проект** = запись в таблице `cae_projects` (id, owner_id, name, created_at, updated_at, current_version_id) + версии модели в `cae_project_versions` (jsonb-снимок).
- **Автосохранение** каждые 30 сек: PATCH `/cae-projects/{id}/draft` (overwrite draft-версии). Полное сохранение версии — по явному действию пользователя.
- **Share-ссылки:** генерируются как nanoid в `cae_project_shares`. Доступ read-only по public link с правом «копировать в свой ЛК».

## 5. Mechanical Engine (концепция, без кода)

### 5.1. Базовый элемент — 3D Timoshenko beam, 2 узла × 6 DOF = 12 DOF

DOF на узел: `[u_x, u_y, u_z, θ_x, θ_y, θ_z]` в **глобальной СК**.
Элементная матрица собирается в **локальной СК** элемента (ось x вдоль оси балки), потом поворачивается матрицей трансформации `[T]` (12×12) в глобальную.

Параметр сдвига для каждой главной плоскости:

```
Φ_y = 12·E·I_z / (G·A_s_y·L²)
Φ_z = 12·E·I_y / (G·A_s_z·L²)
```

Влияние сдвиговой деформации входит как делитель `(1 + Φ)` в члены изгибной жёсткости. При `Φ → 0` Timoshenko ⇒ Bernoulli-Euler.

Структура элементной матрицы `K_e` (12×12) — стандартная блочная: осевая жёсткость `E·A/L`, крутильная `G·I_t/L`, два блока изгиба в плоскостях `xy` и `xz` с поправками Φ_y и Φ_z.

### 5.2. Сборка глобальной матрицы

- Глобальная `K` собирается через DOF-маппинг.
- Внутренние шарниры (`releases`) реализуются через статическую конденсацию соответствующих DOF на уровне элементной матрицы ДО трансформации.
- Внутрипролётные нагрузки сводятся к эквивалентным узловым силам и моментам через `f_eq = ∫ N(x)·q(x) dx`, где `N(x)` — функции формы Timoshenko (см. Reddy «Finite Element Method»).

### 5.3. Граничные условия

Используется **метод исключения уравнений** (вычёркивание строк/столбцов закреплённых DOF) — точнее и быстрее penalty-метода, не вносит численного шума. Для специальных случаев (наклонные роликовые опоры) — multi-point constraints через метод Лагранжа.

### 5.4. Решение

`scipy.sparse.linalg.spsolve` на разрежённой `K` (CSR-формат). Размер задачи в MVP до 5000 элементов × 12 DOF = 60 000 DOF — комфортно укладывается в 256 МБ и 5 сек на cloud function.

### 5.5. Восстановление внутренних усилий

Для каждого элемента:
1. Из глобальных перемещений извлекаем 12 узловых перемещений элемента.
2. Поворачиваем в локальную СК: `u_local = [T]·u_global`.
3. По функциям формы Timoshenko строим эпюры `N(x)`, `Q_y(x)`, `Q_z(x)`, `T(x)`, `M_y(x)`, `M_z(x)` в N точках.
4. Считаем напряжения:
   - `σ_normal(x) = N(x)/A ± M_y(x)/W_y ± M_z(x)/W_z` в характерных точках сечения.
   - `τ_shear(x) ≈ Q(x)·S(x) / (I·b)` или приближённо `Q/A_s`.
   - `σ_eq_VM = √(σ² + 3·τ²)` (плоская задача напряжений в стенке).

### 5.6. Численная стабильность и валидация

- **Smoke-тест:** консольная балка под точечной силой — сравнение с аналитическим решением. Сходимость по сетке.
- **Тест на сдвиг:** короткая консоль (L/h = 5) — Timoshenko должен дать большее перемещение, чем Euler-Bernoulli (на ~5–10%).
- **Тест на 3D-раму:** портал из 3 элементов под боковой нагрузкой — сравнение с SAP2000 / LIRA-САПР референсной моделью.
- **Тест на жёсткое тело:** малая нагрузка, проверка `K·u = 0` при отсутствии нагрузок и удалении 6 DOF (rigid body modes).

### 5.7. Нелинейные расчёты (тариф Профи, с MVP)

**Геометрическая нелинейность (P-Δ):**
- Метод итерационного решения Newton-Raphson с обновлением геометрической матрицы жёсткости `K_geom`.
- Полная касательная матрица `K_T = K_lin + K_geom(u)` пересчитывается на каждой итерации.
- Применение: высокие колонны под сжатием, длинные консоли, элементы под значительной нормальной силой.
- Параметры: `max_iterations` (по умолчанию 30), `tolerance` (1e-4 по норме невязки), `load_steps` (10 шагов нагружения).

**Физическая нелинейность (упругопластичность):**
- Модель материала: билинейная диаграмма σ-ε с упрочнением (Bilinear Kinematic Hardening) или идеальный упругопластический материал (Elastic-Perfectly Plastic) — выбор пользователя.
- Параметры в `material`: `sigma_yield`, `tangent_modulus_E_tan`, `hardening_type` (`bilinear` | `perfect`).
- На уровне сечения — модель пластических шарниров (concentrated plasticity) для рам по нормам Eurocode 3 / СНиП II-23-81* (только для справки, не нормативно).
- Алгоритм: incremental-iterative с return mapping в Гауссовых точках интегрирования сечения.

**Комбинированный режим:**
- В payload `analysis_options.nonlinear.geometric = true` и `physical_plasticity = true` — solver запускает полную нелинейную постановку.
- Время расчёта: задача 100 элементов — 5–15 сек, 1000 элементов — до 60 сек (нужен повышенный таймаут функции, см. §11).

**Валидация:**
- Бенчмарк Эйлеровой устойчивости стойки: критическая сила P_cr должна совпасть с теорией при выходе на нелинейную ветвь.
- Тест пластического изгиба двутавровой балки: момент пластификации `M_pl = σ_y · W_pl` совпадает с аналитикой.

## 6. Frontend-стратегия (R3F)

### 6.1. Структура сцены

```
<Canvas>
  ├── <Grid />               # бесконечная 3D-сетка XY + Z-up
  ├── <Axes />               # глобальные оси
  ├── <Snap />               # snap к сетке, к узлам, к серединам элементов
  ├── <NodesLayer>           # сферы радиуса R, hover/select
  ├── <ElementsLayer>        # цилиндры/линии по 2 точкам
  ├── <BoundaryLayer>        # 3D-символы опор (треугольники, штриховка)
  ├── <LoadsLayer>           # стрелки сил, дуги моментов, эпюры распред. нагрузок
  ├── <SectionVolumeLayer>   # extruded ГОСТ-профили (toggle ON/OFF)
  ├── <ResultsLayer>         # деформированная форма, эпюры N/Q/M, σ_VM heat-map
  └── <TransformControls />  # drag-and-drop узлов (gizmo)
</Canvas>
```

### 6.2. Режимы редактора

- **Draw mode:** клик → добавить узел, второй клик → элемент между узлами. Snap к сетке, к существующим узлам.
- **Edit mode:** выбор узла/элемента → панель свойств справа, gizmo для перемещения узла.
- **Load mode:** клик на узле/элементе → попап выбора типа нагрузки и параметров.
- **BC mode:** клик на узле → выбор типа опоры.
- **Result mode:** read-only, toggle между картой деформаций, эпюрами, картой напряжений.

### 6.3. Снэп и сетка
- Базовый шаг сетки 0.1 / 0.5 / 1.0 м (выбирается).
- Snap к узлу: радиус 8 px на экране.
- Snap к середине / трети / четверти элемента.
- Ortho-mode (Shift): движение только по осям.

### 6.4. Volumetric view

При переключении тумблера каждый элемент рендерится не линией, а 3D-mesh-ом, полученным экструзией 2D-контура своего ГОСТ-сечения вдоль вектора элемента. Это позволяет визуально проверить корректность ориентации главных осей и собираемости узлов.

### 6.5. Модульная система отчёта

`ReportComposer` — компонент с чекбоксами:

- [x] Исходная геометрия (таблицы узлов, элементов)
- [x] Материалы и сечения
- [ ] Глобальная матрица жёсткости (для образовательных целей)
- [x] Граничные условия и нагрузки
- [x] Максимальные перемещения и реакции
- [x] Эпюры N, Q, M
- [x] Карта эквивалентных напряжений
- [x] Выводы и коэффициенты запаса
- [ ] Приложение: полные данные узлов в CSV

Кнопка «Сформировать» → POST на `/cae-report` с массивом выбранных секций → бэкенд через **Jinja2** рендерит HTML по шаблонам, **WeasyPrint** конвертирует в PDF с правильной поддержкой кириллицы и формул (MathJax → SVG до экспорта).

Шаблон PDF в стиле пояснительной записки ВКР: рамка ГОСТ 2.104-2006, Times New Roman 14pt, нумерация формул, библиография в конце.

## 7. Биллинг и интеграции

- **Платёжный шлюз:** Robokassa или Юкасса (оба российские, есть готовое расширение poehali.dev).
- **Подписка:** ежемесячная автоматическая через рекуррентные платежи.
- **Разовый расчёт:** одноразовый чек, доступ к 1 проекту на 7 дней.
- **Битрикс24:** при нажатии «Нужна помощь наставника?» — webhook в Битрикс с заполненными полями `user_email`, `project_name`, `n_elements`, `тариф`.
- **Email через Яндекс 360 SMTP:** уведомления о платежах, о завершении расчёта, о приближении срока подписки.

## 8. Безопасность

- HTTPS only, HSTS, CSP, CORS на whitelist.
- JWT signed RS256, секрет в env-переменных Cloud Functions.
- Все пароли — Argon2id, salt + pepper.
- Rate-limit на solver и report (Redis-токены или PostgreSQL counters).
- Аудит-лог критичных операций (login, payment, project_delete).
- Защита от XSS в публикуемых PDF (sanitization комментариев пользователя).
- Защита от DoS: размер payload ≤ 2 МБ, число элементов ≤ tariff_limit.

| № | Вопрос | Решение | Влияние на план |
|---|--------|---------|------------------|
| 1 | Нелинейные расчёты | **Полный пакет в Профи сразу:** P-Δ (геометрическая) + упругопластичность (билинейное упрочнение + perfect-plastic) | M10, +4 нед. См. §5.7 |
| 2 | Командный доступ Профи | **Owner / Editor / Viewer** (стандарт индустрии — Figma, Notion) | M9, +1 нед. БД-таблица `cae_project_members` с полем `role` |
| 3 | STEP-импорт | **Упрощённый парсер на чистом Python** в M11. Без pythonocc, без VPS. Парсим `CARTESIAN_POINT`, `EDGE_CURVE`, `VERTEX_POINT` из STEP AP203/AP214 → wireframe сетка узлов и элементов. Достаточно для рам, не работает с B-Rep/поверхностями. | M11, +1,5 нед. |
| 4 | Сейсмика / динамика / строители | **НЕ делаем.** Фокус продукта — машиностроение. Решение о выделении строительного направления — после MVP по результатам обратной связи. | Убрано из Roadmap |
| 5 | Сертификация | **Обучающий + инженерный инструмент.** Без аттестации Росстандарта и ЯБОК. Дисклеймер во всех PDF-отчётах: «Расчёт выполнен для учебных и предварительных инженерных целей. | 7 | Публичный API | **Отложен до v2.0.** В MVP — только внутренний API между фронтом и бэком (JWT). Публичный API + OAuth — после валидации продукта (100+ платящих). | Не влияет на MVP, освобождает 1+ нед. |

## 12. Дисклеймер для отчётов (юридическая защита)

Текст, который автоматически вставляется в шапку каждого PDF-отчёта:

> **Внимание.** Настоящий расчёт выполнен в облачной CAE-системе «Диплом-Инж.CAE» и предназначен для учебных целей и предварительной инженерной оценки. Программное обеспечение не аттестовано Росстандартом и не имеет статуса средства измерений или поверочного расчётного комплекса. Окончательные проектные решения должны быть проверены аттестованным расчётным ПО (LIRA-САПР, SCAD, ANSYS, APM WinMachine) и согласованы с ответственным конструктором/нормоконтролёром организации-исполнителя.

Дисклеймер не отключается. На демо-расчётах добавляется водяной знак «DEMO» на каждой странице.
